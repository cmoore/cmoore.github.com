1 The 'what'

Brazil is hybrid framework for developing web applications. It combines an application server with a database, templating, and delivery management API.

1.1 Philosopy

  1. The 'one second' rule.

    As more potential visitors have access to faster internet connections, the 'one second' rule becomes more and more important. The rule states that you have one full second to capture a visitor's attention. After that, you get into the realm of impatience, and no one wants that. Brazil aims to be the fastest available web framework several reasons, chief among them being that while we are very fast, other components in your setup may not be. Without the need for external data, Brazil can parse a template and serve the page in sub-milliseconds. Database access is arguably the most costly operation you can do in a web application and eventually, Brazil will include an automatic eventual-consistency cache that is shared across all of its instances, but for now we have to be a fast as we can to make up for the time spent fetching data. Even if your database is comparatively slow, you might just appear quite snappy to your visitors. This is the hope, anyway.

1.2 Key Points

  1. Applications are strictly type-safe. ie. a string can never be mistaken for a number. Ever.
  2. Uses the functional programming language 'Haskell'.
    • Haskell is hard, but we make it easy to get the best of both worlds.
      • Strictly typed function programming.
      • Rapid development.
  3. Concurrency is built in. It will use as many CPU cores as you care to give it.
    • The concurrency implementation is 20 years old.
  4. Built to scale
    • Compiles down to a binary that is designed to respond to proxied web requests.
    • Run a site on 1 or 1000 instances of Brazil. It won't stomp on itself creating weird errors for your visitors.

2 The 'how'

Brazil provides quite a lot of specialized functionality in the form of modules. The best analogy is to consider the UNIX command line environment. Lots of small tools that do one or two very specific things and produces and consumes data in the same format so that commands can be easily composed to form larger, more specialized commands.

2.1 The API

2.1.1 Core Types

While it would be a good idea to understand these types, it is not required to use Brazil. There are utility functions to handle most common situations in the API.

  1. BRequest - Contains lower level information about the request itself.

2.1.2 Function Utilities

  • Requests
    • rarg - (rarg rq) : For a given POST'ed argument, return the value. Example: 
    1:  <form method=POST ...>
2:  <input type="text" name="title"/>
3:  </form>

    1:  -- Where 'rq' is the request object passed to your handler.
2:  my_handler rq = do
3:    let title = rarg "title" rq
    • renv - (renv rq) : A list of the arguments passed. The list has the form [(name,value)].
    • rrtype - (rrtype rq) : The type of the request. eg. 'GET', 'POST', etc. The value is a uppercase constant. 
    1:  handler rq = do
2:    case rrtype rq of
3:      'GET' -> startT "error" >>= finishT
4:      'POST' -> ...

3 Reference

  • Templates
    Brazil templates follow the StringTemplate design.
    • API
      • Setting up the data 
      1:  view_posts :: BRequest -> IO Response
2:  view_posts rq = do
3:    posts <- read_latest_posts
4:    user <- get_current_user rq
5:    startT rq "posts" >>=
6:     setAttr "posts" posts >>=
7:     setAttr "user" user >>=
8:     finishT

      In this example, startT sets up the template, setAttr binds "posts" to the data inside post and likewise user. Finally, finishT compiles the template down and does the final substitutions, then prepares the response for the browser and sends it.

      • Viewing the data 
       1:  <div>
 2:    $if(user)$
 3:    <span>Hello, $user.name$
 4:    $endif$
 5:  </div>
 6:  
 7:  $if(posts)$
 8:  <table>
 9:  $posts:{ p |
10:    <tr><td><b>$p.title$</b> posted on $p.date$</td></tr>
11:    <tr><td colspan=2>$p.body$</td></tr>
12:    <tr><td>Posted by: $p.author$</td></tr>
13:  }$
14:  </table>
15:  $endif$

      Here is an example of both single element arguments, and a compact, interator-style use of template arguments. posts is assumed to be a list of posts.

  • KeyValue
    A convenient abstraction on top of storing a retrieving key/value pairs. Here is a running list of all of the available functionality and how to use it.
    • Types 
      1:  data Kv = Kv {
2:    kv_id :: Maybe Int,
3:    kv_key :: String,
4:    kv_val :: String
5:  } deriving( Show )
    • API 
      1:  let b = Kv {
2:           kv_id = Nothing,
3:           kv_key = "honk",
4:           kv_val = "wonk"
5:          }
      1. get 
        1:  get "honk"
      2. set 
        1:  ba <- set b

        Commit this value to the database. 

        1:  bb <- set $ ba { kv_key = "honka" }

        Commit the element again, but this time with a new value property. 

        1:  get $ kv_key ba -> Nothing

        Now the original is not in the database because updating a row in the database increments its id number (kv_id).

  • Sonar - automatic clustering
    An abstraction against detecting other instances of Brazil on the same subnet and forming a cluster automatically.
  • IPUBSUB - internal pub/sub
    Brazil provides an abstraction for inter-process communication via ZeroMQ.
